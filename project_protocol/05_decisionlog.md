# 决策记录

## 1. 主系统决策
决定继续由 OpenClaw 担任项目主系统。原因是项目的核心价值仍然在于跨步骤的调度、判断、评分、写作编排、审核交付与留档，而不是单点采集或单次写作。

## 2. 原项目复用决策
决定继续轻量复用 BettaFish 的真实输入能力与既有报告撰写能力，而不把其完整系统边界整体并入 OpenClaw。原因是下一步目标是补齐用户可用运行面，而不是重写输入系统与报告系统。

## 3. 双输入模式决策
决定正式采用双输入模式作为统一需求与统一实现口径：
- 一类输入来自真实来源或热点流；
- 一类输入来自用户手动输入主题、公司、赛道或关键词。

原因是两类场景都是真实使用场景：
- 来源驱动场景适用于日常扫描、热点发现与自动候选输入；
- 主题驱动场景适用于专题策划、指定公司跟踪与编辑主动选题。

## 4. 双输入模式统一主链路决策
决定无论输入来自真实来源还是用户主题，都必须汇入同一条后续主链路，而不建立两套平行评分、写作、交付协议。原因是项目真正需要统一的是选题、评分、写作、交付与留档，而不是让不同输入方式演化为不同系统。

## 5. 用户主题输入边界决策
决定将用户主题输入定义为围绕主题发起抓取与候选整理的输入方式，而不是用户直接指定最终选题。原因是用户给出主题后，系统仍需经过抓取、聚合、选题、评分，才能保证证据门槛与状态分流不被绕过。

## 6. 统一编排内补齐能力决策
决定继续以 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 为唯一主编排核心，在统一编排内部补齐双输入、显式选题、分阶段执行与继续执行能力，而不是再创建一套平行顶层编排。原因是：
- 现有统一编排已经具备稳定的结果收口、降级与留档语义；
- 新能力本质上是对现有主链路的补齐，而不是替换主链路；
- 在单一主编排内扩展更容易维持 `entry_options`、`entry_resolution`、阶段状态与回放对象的一致性。

## 7. 总启动入口与统一编排分层决策
决定继续区分“内部统一编排核心”与“面向真实使用者的总启动门面”两个层次：
- [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 继续作为当前已经成立的统一编排核心；
- 更面向真实使用者的唯一推荐总启动门面仍保留为后续产品化收口事项。

原因是当前已经完成的是内部主链路与执行模式补齐，而不是最终的外层使用门面；如果把这两层混写成一个已完成事实，会导致需求、计划与阶段材料口径冲突。

## 8. 显式选题阶段决策
决定将选题层从抓取与评分之间显式提取出来，形成独立产物，而不是继续让候选事件直接跳入评分。原因是编辑工作流天然需要先看选题，再决定是否继续评分与写作；没有显式选题层，系统就无法真正服务编辑选择。

## 9. `topics` 阶段继续在统一编排内生成决策
决定由统一编排直接组织 [`build_topic_cards()`](clawradar/topics.py:273) 与相关工件，而不是把选题层拆成独立平行入口。原因是选题阶段既需要承接上游输入模式差异，又需要给下游评分、写作、交付提供统一审核对象；把它保留在主编排中，更能保证全链路状态一致。

## 10. 运行模式矩阵决策
决定统一编排必须同时支持全流程运行与分阶段运行，而不是只保留统一全链路执行。原因是真实使用场景需要支持：
- 只抓数据；
- 只出选题；
- 只评分；
- 只写作；
- 只交付；
- 从已有结果继续执行。

## 11. `write_only` 补齐路径决策
决定通过复用已有 `scored_events` 与写作阶段输入组装逻辑，在统一编排内补齐 `write_only`，而不是单独设计一套新的写作入口协议。原因是写作阶段本身依赖统一编排已经收口的评分结果、目标事件与写作配置；在主编排内补齐可直接复用状态回写、降级与结果收口。

该决策最终落地为 [`_build_write_payload()`](clawradar/orchestrator.py:714) 与 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 内的 `write_only` 分支。

## 12. `resume` 补齐路径决策
决定通过识别现有阶段工件来推断最近完成阶段，并继续向下执行，而不是要求调用方重新手工拼接完整顶层请求。原因是真实使用者更关心“已有哪类工件，下一步该怎么继续”，而不是重新理解整套内部协议。

该决策最终落地为 [`_resolve_resume_target()`](clawradar/orchestrator.py:703) 的恢复目标推断规则：
- 有 `content_bundles` 时优先继续到交付；
- 有 `scored_events` 时优先继续到写作；
- 有 `topic_cards` 或 `normalized_events` 时优先继续到评分；
- 否则回到主流水线起点。

## 13. 阶段工件继续执行决策
决定继续执行能力建立在现有阶段工件之上，而不是建立在隐式上下文或外部记忆之上。原因是只有可显式传递与可回放的工件，才能同时支持：
- 局部重跑；
- 失败恢复；
- 审核核对；
- 运行留痕与回放。

因此，本轮继续沿用并强化 `crawl_results`、`topic_cards`、`scored_events`、`content_bundles`、`delivery_receipt`、`run_summary` 等结构化结果对象，而不引入新的平行结果协议。

## 14. 双输入收口方式决策
决定把 `user_topic`、`real_source`、`inline_topic_cards` 等输入方式都统一纳入 `entry_options.input.mode` 的解析框架，由 [`_build_entry_resolution()`](clawradar/orchestrator.py:236) 与 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 统一决议。原因是：
- 统一入口配置已经是当前最稳定的内部协议；
- 输入差异主要体现在上游工件准备，而不是下游评分、写作、交付协议；
- 收口到单一解析框架，更有利于 fallback、回放与事实追踪。

## 15. 已完成事实与后续事项分层决策
决定在协议文档中显式把“阶段十基线”“本轮已完成实现”“后续待补齐事项”分层记录，而不是只保留单一时间点口径。原因是当前项目已经同时存在：
- 阶段十留下的历史证据链；
- 本轮在代码与测试层已经完成的新能力；
- 尚未完成的产品化门面与说明材料。

如果不分层记录，容易出现两类错误：
- 把阶段十材料误写成已经证明本轮新增能力；
- 把仍待后续补齐的事项误写成当前已经完成。

## 16. `01_requirements.md` 与 `02_constraints.md` 保持克制更新决策
决定本轮优先更新 [`project_protocol/03_plan.md`](project_protocol/03_plan.md)、[`project_protocol/04_changelog.md`](project_protocol/04_changelog.md)、[`project_protocol/05_decisionlog.md`](project_protocol/05_decisionlog.md) 与阶段十说明材料；对 [`project_protocol/01_requirements.md`](project_protocol/01_requirements.md) 与 [`project_protocol/02_constraints.md`](project_protocol/02_constraints.md) 暂不进行大幅改写，仅在必要时做后续澄清。

原因是：
- [`project_protocol/01_requirements.md`](project_protocol/01_requirements.md) 仍主要负责描述“要做什么”；
- [`project_protocol/02_constraints.md`](project_protocol/02_constraints.md) 仍主要负责描述“不能怎么写、不能怎么做”；
- 当前更需要修正的是计划、变更、决策与阶段十材料对现实状态的表达，而不是重写上位需求与约束。

## 17. skill 统一总启动门面定位决策
决定将 skill 定位为“统一总启动门面的外层调用载体”，而不是新的平行主入口实现。原因是：
- [`project_protocol/02_constraints.md`](project_protocol/02_constraints.md) 已明确要求继续保持统一主入口口径，不能扩写多个互相冲突的顶层入口；
- 当前已经稳定成立的主编排核心仍是 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)，skill 更适合作为面向调用方的轻量外层门面，而不是替代内部统一编排；
- 以 skill 承接外层调用，可以在不破坏现有主链路与回放语义的前提下，补齐用户可见的启动方式与说明材料。

## 18. skill 必须继续收口到统一入口决策
决定 skill 在 MVP 与后续方案中都必须继续收口到统一入口 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)，而不是各自直连分散阶段函数或另建外层编排。原因是：
- 统一入口已经承接 `entry_options`、`entry_resolution`、阶段跳过、降级与最终结果收口语义；
- 如果 skill 直接绕开统一入口，将破坏双输入、分阶段执行、继续执行与留档回放的一致性；
- 统一收口可以避免 skill 资产演化成新的顶层协议，从而触犯“不得制造平行顶层入口”的约束。

## 19. MVP 不先做包装脚本决策
决定本轮 MVP 不优先新增包装脚本，而是先让外层 skill 直接调用 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)。原因是：
- 本轮最小目标是补齐“可由 skill 调用项目”的最小方案，不是新增另一层可执行壳；
- 包装脚本会增加新的调用协议、参数同步与维护面，反而容易造成外层门面与统一入口口径漂移；
- 在统一入口已存在的前提下，先以 skill 直接收口可以更快完成最小交付，并把包装脚本保留为后续可选增强。

## 20. 本次最小交付范围决策
决定本次最小交付范围以 skill 资产、项目文档、协议同步为主，而不把包装脚本、额外壳层或附加演示包预先写成必做事实。原因是：
- 当前最缺的是调用收口方式与说明材料，而不是再次扩充新的执行入口；
- 先补齐 skill 资产与文档，才能让后续使用文档、验收口径与项目协议保持一致；
- 将范围压到最小，有利于避免把尚未实现的附加增强项提前写成既成事实。

## 21. 阶段十材料不能承担本轮 skill 方案证明决策
决定不把 [`project_protocol/stage10/`](project_protocol/stage10/) 下材料作为本轮 skill 方案的证明材料。原因是：
- 阶段十材料证明的是统一编排基线、最小留档与历史演示验收收口已经成立，不包含 skill 作为外层调用载体的设计与使用事实；
- 即使统一入口 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 已在阶段十基线中成立，也不能倒推出“skill 调用方案已经被阶段十证明”；
- 若把阶段十改写成 skill 方案证明，会把历史基线材料误用为新能力证据，违背协议的事实分层要求。

## 22. `04_changelog.md` 延后回写决策
决定 [`project_protocol/04_changelog.md`](project_protocol/04_changelog.md) 必须等 skill 资产、项目文档或其他实际落地内容真实完成后，再回写对应事实，而不是在当前仅有计划与决策更新时提前记为变更完成。原因是：
- [`project_protocol/04_changelog.md`](project_protocol/04_changelog.md) 只记录已改内容，不记录尚未落地的方案设想；
- 当前子任务范围仅限计划层与决策层，同步到变更事实层会造成“未实现先记账”；
- 延后回写可以确保变更记录继续作为真实落地事实的索引，而不是混入预期性描述。

## 23. 阶段十材料与本轮实现关系表述决策
决定在后续需要补充关系说明时，仍只把 [`project_protocol/stage10/`](project_protocol/stage10/) 视为历史基线材料，不回填为 skill 新方案证据。原因是历史材料需要与当前现实状态建立联系，但又必须保留历史证据边界，避免出现“旧证据证明新方案”的事实越界。

## 24. 非平台化决策
决定即使补齐双输入、显式选题、分阶段执行、继续执行与 skill 外层调用载体，也不把项目扩写为完整平台后台、自动发布平台或无人审核写稿系统。原因是项目边界仍应保持在选题、评分、写作编排、审核交付与留档这一条主工作流之内。

## 25. 本轮完成判断决策
决定将以下内容记录为本轮已完成事实：
- 双输入模式中的 `real_source` 与 `user_topic` 已统一收口到主链路；
- 显式选题阶段已经成立；
- `crawl_only`、`topics_only`、`score_only`、`write_only`、`deliver_only`、`resume` 已在统一编排中成立；
- 基于阶段工件继续执行的基础能力已经成立；
- 协议中的计划层与决策层已补写 skill 最小方案的定位、边界与延后回写原则。

同时决定将以下内容继续记录为后续事项：
- skill 资产、项目文档与使用说明的真实落地；
- 如有必要再补充包装脚本等外层增强；
- 新增能力对应的成套演示、运行说明与验收材料；
- 更进一步稳定的目录规范、回放手册与体验层增强。
