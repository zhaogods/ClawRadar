# 项目约束

## 1. 文档职责
定义项目边界、禁止事项、表达规范与后续补齐方向的约束条件，确保需求、计划、变更、决策不会相互混淆。

## 2. 范围约束
- 不将项目定义为完整舆情平台
- 不将项目定义为全自动无人审核写稿系统
- 不扩展为公众号自动发布平台
- 不将 OpenClaw 部署本身作为本项目核心目标
- [`project_protocol/stage10/`](project_protocol/stage10/) 材料只证明统一编排基线、最小演示留档与历史验收收口已成立，不等于当前新增实现与后续产品化运行形态已经全部由阶段十材料覆盖
- 当前允许继续扩展的范围限于：总启动门面、统一使用说明、标准工件说明、运行目录说明、演示与验收材料补齐，以及对现有双输入、显式选题、分阶段执行、恢复执行能力的文档化收口
- 不将本轮协议扩写为平台后台、权限系统、运营看板或批量自动发布系统

## 3. 架构约束
- 原项目真实采集层与原项目报告撰写层都只能作为被调用能力，不得成为主系统
- OpenClaw 必须持续承担主调度、主判断、主评分、主审核交付与主留档职责
- 总启动门面必须是统一门面，而不是一套平行实现；其内部仍应复用现有统一编排链路与阶段能力
- 双输入模式必须汇入同一主链路，不得拆成两套平行评分、写作、交付协议
- 用户主题输入是一种新增输入方式，不得把它写成替代真实来源输入的唯一模式
- 真实来源输入仍是合法主输入之一，不得因为引入用户主题输入而降级为历史兼容能力
- 写作阶段仍应优先调用原项目既有报告撰写能力；OpenClaw 不直接承担最终长文生成主体职责
- 原项目报告撰写能力仅作为写作执行层，不得替代 OpenClaw 的选题判断、流程控制、失败分流、审核交付与留档

## 4. 运行模式约束
- 项目必须保留统一主入口口径，不得新增多个互相冲突的顶层入口
- 当前已落地的执行模式可以在协议中按既成事实引用，但不得据此推出“总启动门面说明、运行手册、阶段材料”也已全部完成
- 分阶段运行至少覆盖抓取、选题、评分、写作、交付与恢复运行
- 分阶段运行必须建立在标准中间工件之上，不得要求用户手工拼接复杂 JSON 才能继续执行
- 运行模式的产品化表达可以更简化，但内部仍需与 `entry_options`、`entry_resolution`、既有触发语义兼容
- `manual`、`cron`、`single_event_rerun` 语义必须继续兼容，不得被新的产品化运行模式破坏

## 5. 数据与内容约束
- 所有重要结论必须有来源支撑
- 无法确认的信息必须标注不确定性
- 弱证据、泛趋势、纯情绪类主题不得直接进入正式长文写作阶段
- 允许输出 `watchlist`、`need_more_evidence` 或 `no_publish`，不强行生成弱稿
- 真实来源输入与用户主题输入都必须遵守同样的证据门槛与状态分流规则
- 用户主题输入不等于用户直接指定最终选题；主题输入后仍需经过抓取、聚合、选题、评分流程
- 选题阶段必须形成可审核产物，不得把“候选事件直接评分”继续作为唯一对外运行形态
- 真实交付面只能承接审核友好内容，不得把未经审核内容表达为可直接自动外发结论
- 交付回执必须保持可追溯，不能只剩不可回放的外发结果描述

## 6. 实现约束
- 优先轻量集成原项目的真实采集能力与既有报告撰写能力，不将其完整系统边界一并并入 OpenClaw 主职责
- 优先使用结构化中间产物，避免仅靠自由文本串联流程
- 必须补齐抓取结果、选题结果、评分结果、写作结果、交付结果等阶段工件，不得只依赖顶层总结果对象
- 总启动门面应对用户暴露少量核心参数，不得把底层复杂协议直接暴露为主要使用方式
- 写作请求必须以结构化字段驱动，不得把 OpenClaw 改造成直接承接超长正文生成的大提示词写稿器
- 对外协议、文档与后续规划都必须以统一入口作为唯一主入口口径，不再扩写平行主入口或平行顶层选择表达
- 统一入口的内部配置表达继续优先收口为 `entry_options` 下的 `input`、`write`、`delivery`、`degrade` 四类选择，不再扩写平行顶层选择口径
- 当旧字段与统一入口配置并存时，文档、协议与实现说明都必须以 `entry_options` 为优先口径，不得把冲突配置写成并列生效
- 统一结果必须保留 `entry_resolution` 一类集中回放对象，用于说明本次实际采用的入口选择、降级策略与 fallback 结果，不得要求调用方依赖分散字段自行反推
- 配置、密钥、渠道、触发方式必须可替换，不写死为单一环境或单一渠道
- 当写作被显式关闭时，流程必须允许停在 score 或 topics 后结束，不得强制进入写稿阶段
- 当交付目标为 `archive_only` 时，必须真实落地本地留档并返回可追溯回执，不得仅保留名义留档语义，也不得将其表述为已触发真实外发
- 当外部交付渠道不可用且声明按 `archive_only` 降级时，必须回退为本地留档并保留回退回执，不得静默跳过交付，也不得丢失审核所需留档信息
- 对不可用 `real_source`、`external_writer` 或不可用交付渠道，必须显式失败或按声明的降级策略回退，不得静默忽略
- 统一入口收口只表示入口选择、优先级、回放与降级语义被统一，不得把该收口表述为真实采集、真实 external writer 或真实自动发布已经全部接通之外的额外事实
- 运行方式必须支持本地执行与受控触发，不将云端常驻部署视为前置条件

## 7. 文档与事实约束
- [`project_protocol/`](project_protocol/) 下的协议文档可以新增下一步目标，也可以回写已核实的实现事实，但必须明确区分阶段十历史基线、本轮已完成实现与后续待补齐事项
- 阶段十材料只能继续作为历史证据包与当前基线说明，不得被改写为已经覆盖双输入模式、显式选题阶段、分阶段运行、`write_only`、`resume` 等本轮新增能力的证明
- [`project_protocol/01_requirements.md`](project_protocol/01_requirements.md) 只写要做什么
- [`project_protocol/03_plan.md`](project_protocol/03_plan.md) 只写怎么推进，以及哪些计划项已经转化为事实
- [`project_protocol/04_changelog.md`](project_protocol/04_changelog.md) 只记已改内容
- [`project_protocol/05_decisionlog.md`](project_protocol/05_decisionlog.md) 只记为何如此决定
- 所有新增表述都必须使用阶段十基线、已完成事实、后续要求、计划、约束、决策等与当前阶段相符的口径，不得把尚未形成说明材料、运行门面或验收材料的内容写成已经整体完成
