## 3. MindSpider 分析

### 3.1 原有定位

`MindSpider` 是一套围绕"热点发现 + 内容深爬"的采集型引擎，包含两层：

1. `BroadTopicExtraction` — 面向 12 个来源榜单的热点抓取与话题提取
2. `DeepSentimentCrawling` — 面向 7 个社媒平台的内容深爬（依赖 MediaCrawler 子模块）

### 3.2 当前主线使用情况

主线通过 `clawradar/real_source.py` 动态导入，升级后接入：

| 功能 | 入口 | 状态 |
|------|------|------|
| `NewsCollector.get_popular_news()` | `_collect_mindspider_news()` | ✅ 改造前已接入 |
| `DatabaseManager` 持久化 | `_collect_mindspider_news(persist=True)` | ✅ P2 新增 |
| `collector.collect_and_save_news()` | 同上 | ✅ P2 新增 |
| `TopicExtractor.extract_keywords_and_summary()` | 同上 | ✅ P2 新增（AI 关键词提取） |
| `DeepSentimentCrawling.run_daily_crawling()` | `_run_deep_sentiment_crawling()` | ✅ P2 新增（编排器 DEEP_CRAWL 阶段） |
| `KeywordManager` — 从 DB 取关键词 | DeepSentimentCrawling 内部调用 | ✅ P2 接入 |
| `PlatformCrawler` — 调 MediaCrawler | DeepSentimentCrawling 内部调用 | ✅ P1 初始化 |

### 3.3 当前实际承担的角色

- `real_source` 模式：12 源榜单热点聚合 → 可选 DB 持久化 + AI 关键词提取
- `deep_crawl` 阶段：评分后触发 DEEP_CRAWL，从 DB 取 BroadTopic 关键词 → 7 平台社媒深爬 → 证据注入

### 3.4 当前未被主线直接用到的能力

- `main.py` 完整 CLI（`--status` / `--complete` 等交互模式）
- `schema/init_database.py` 数据库初始化脚本
- MediaCrawler 内部 FastAPI 接口（`api/`）
- MediaCrawler 内部 IP 代理池（`proxy/`）
- MediaCrawler 内部测试文件

### 3.5 结论

MindSpider 从"热点发现层"升级为"热点发现 + DB 持久化 + AI 关键词 + 社媒深爬"的完整采集引擎。深爬默认关闭，需通过 `entry_options.deep_crawl.enabled=true` 显式开启。

## 4. QueryEngine 分析

### 4.1 原有定位

`QueryEngine` 是面向新闻和信息检索的搜索引擎，围绕 Tavily 工具集封装了 6 种粒度的搜索能力：

| 能力 | 定义位置 |
|------|---------|
| `basic_search_news` | `tools/search.py` |
| `deep_search_news` | `tools/search.py` |
| `search_news_last_24_hours` | `tools/search.py` |
| `search_news_last_week` | `tools/search.py` |
| `search_images_for_news` | `tools/search.py` |
| `search_news_by_date` | `tools/search.py` |

### 4.2 当前主线使用情况

主线通过 `clawradar/real_source.py` 动态导入，升级后接入：

| 功能 | 入口 | 状态 |
|------|------|------|
| `search_news_last_week(query)` | `_search_with_tavily()` | ✅ 改造前已接入 |
| `search_news_last_24_hours(query)` | `_search_with_tavily()` | ✅ P3 新增（多窗口并发） |
| `search_images_for_news(query)` | `_search_with_tavily()` | ✅ P3 新增（图片搜索） |
| 深度搜索管线 `DeepSearchAgent.research()` | 未接入 | ❌ |

### 4.3 当前实际承担的角色

- `user_topic` 模式：作为**第一搜索提供商**（tavily_news），执行双窗口（24h + 7d）并发搜索 + 图片搜索
- 搜索结果按时间衰减权重合并（24h=1.0, 周=0.7），24 小时内新闻优先

### 4.4 当前未被主线直接用到的能力

- `deep_search_news` — 深度新闻分析
- `search_news_by_date` — 按日期范围搜索
- `QueryEngine.agent` 完整 Agent 推理链（4 阶段：plan→search→reflect→format）

### 4.5 结论

QueryEngine 从"一周搜索工具"升级为"多窗口 + 图片搜索的增强搜索 provider"。完整 DeepSearchAgent 管线仍未接入（需较大改造量，非当前优先级）。

## 5. MediaEngine 分析

### 5.1 原有定位

`MediaEngine` 是多模态搜索与媒体检索引擎，提供两套搜索类：

- `BochaMultimodalSearch` — web/news/images/structured_data 多通道
- `AnspireAISearch` — web/news 搜索

### 5.2 当前主线使用情况

主线通过 `clawradar/real_source.py` 动态导入，升级后接入：

| 功能 | 入口 | 状态 |
|------|------|------|
| `BochaMultimodalSearch.search_last_week(query)` | `_search_with_bocha()` | ✅ 改造前已接入 |
| `BochaMultimodalSearch.search_last_24_hours(query)` | `_search_with_bocha()` | ✅ P3 新增 |
| `BochaMultimodalSearch.search_for_structured_data(query)` | `_search_with_bocha()` | ✅ P3 新增 |
| `AnspireAISearch.search_last_week(query)` | `_search_with_anspire()` | ✅ 改造前已接入 |
| `AnspireAISearch.search_last_24_hours(query)` | `_search_with_anspire()` | ✅ P3 新增 |

### 5.3 当前实际承担的角色

- `user_topic` 模式：作为**第二、第三搜索提供商**（bocha_search + anspire_search）
- Bocha：双窗口（24h+7d） + 结构化数据（股票/天气/百科等）
- Anspire：双窗口（24h+7d）
- 三提供商结果通过 `_round_robin_take` 融合去重

### 5.4 当前未被主线直接用到的能力

- `MediaEngine.agent` 完整 Agent 流程（节点/状态/prompts）
- 更多多模态搜索工具组合

### 5.5 结论

MediaEngine 从"第二搜索通道"升级为"双窗口 + 结构化数据的完整多模态搜索 provider"。完整 Agent 管线未接入。

## 6. ReportEngine 分析

### 6.1 原有定位

`ReportEngine` 是四个引擎中最重的一层，负责完整的报告生成管线：

- LLM 客户端 → 模板/章节规划 → 章节生成 → IR 装订 → HTML/Markdown/PDF 渲染 → 校验修复

### 6.2 当前主线使用情况

主线通过 `clawradar/writing.py` 动态导入：

| 功能 | 入口 | 状态 |
|------|------|------|
| `create_agent()` / `ReportAgent.generate_report()` | `_get_report_engine_agent_factory()` | ✅ 作为 external_writer 正式执行器 |

### 6.3 当前实际承担的角色

- 接收 `query + reports + forum_logs + custom_template`
- 生成 HTML / IR / 状态文件等写作产物
- 返回 `writer_receipt` 和报告落盘路径

### 6.4 当前未被主线直接用到的能力

- Flask 接口（`flask_interface.py`，1452 行）
- 图表修复与审查链（`utils/chart_repair_api.py` 等）
- PDF 导出链（`scripts/export_to_pdf.py`）
- 旧多引擎报告聚合兼容逻辑

### 6.5 结论

ReportEngine 是四引擎里"最完整被复用"的一个，无需变动。

## 9. 对后续阶段的建议

### 9.1 MindSpider

- 深爬触发条件可进一步细化（当前仅按 `publish_ready` 决策触发，可加关键词热度阈值）
- `_filter_keywords_by_platform()` 平台偏好过滤可集成到深爬流程

### 9.2 QueryEngine / MediaEngine

- 当前三提供商双窗口搜索方案已满足需求
- `search_news_by_date()` 月度趋势可按需快速接入
- 不建议接入完整 Agent 框架（维护成本 > 收益）

### 9.3 ReportEngine

- 保持现状，不做变更
- Flask 接口、PDF 导出等旧能力继续保持"保留不删"策略

## 10. 最终结论

深度爬取增强（2026-04-30）完成后的四引擎状态：

- **MindSpider**：从"15%热点采集层"升级为"60%完整采集引擎"，DB 持久化 + AI 关键词 + 7 平台社媒深爬已全部接回主线；
- **QueryEngine**：从"10%一周搜索"升级为"40%多窗口+图片搜索"，24 小时时效新闻和封面素材可用；
- **MediaEngine**：从"10%一周搜索"升级为"40%多窗口+结构化数据"，股票/天气/百科等结构化信息可用；
- **ReportEngine**：保持 70% 利用率为最高，写作管线满足当前需求。

主链路从"浅层热点聚合 → 单次搜索"升级为"热点发现 → AI 关键词提取 → 7 平台社媒深爬 → 三提供商多窗口多模态搜索 → 证据包注入 → 评分 → 写作 → 投递"的完整深度爬取管道。
