# WeChat Image Strategy Plan

## Background

The current WeChat draft publishing flow in `clawradar/publishers/wechat/` has already separated responsibilities into:

- `service.py`: delivery orchestration only
- `report_html_cleaner.py`: report HTML extraction and WeChat-safe simplification
- `markdown_converter.py`: Markdown-to-WeChat HTML conversion
- `image_handler.py`: media policy routing

At the moment, the channel supports four image policies:

- `drop`
- `placeholder`
- `fallback_table`
- `upload`

For the latest real report used in publishing, the effective structure is:

- `img = 0`
- `canvas = 8`
- `svg = 3`
- `chart-fallback = 8`

This explains the current behavior:

- `upload` already works for real `<img>` nodes
- the latest report still shows no images in WeChat because it contains chart canvases rather than inline `<img>`

This document excludes the browser screenshot path on purpose and compares only two practical chart-image strategies.

## Upstream Clues From ReportEngine

The report pipeline already exposes two useful properties:

1. `html_renderer.py` emits chart widgets as `canvas + chart-fallback`
2. `chart_to_svg.py` already exists and can convert Chart.js-like widget payloads into static SVG

That means WeChat chart-image support does not have to start from scraping a running browser. The upstream report engine already provides both:

- a text/table fallback representation
- a static chart rendering direction

## Goal

Introduce a stable WeChat article image strategy for charts without relying on headless-browser screenshots.

The solution should:

- preserve the current safe default publish path
- avoid putting heavy rendering logic into `service.py`
- keep the WeChat draft workflow review-friendly
- remain compatible with future multi-channel publishing

## Option 1: Chart Fallback Table To PNG

### Idea

Read each `.chart-fallback table` from the report HTML, render the table into a static PNG locally, upload the PNG through WeChat `media/uploadimg`, then replace the chart block with `<img>` in the final WeChat article body.

### Data Source

- Source: `html_renderer.py` output HTML
- Required input: `.chart-fallback table`
- No dependency on front-end JavaScript execution

### Expected Pipeline

1. `report_html_cleaner.py` extracts article HTML as it does today
2. `image_handler.py` detects chart containers in `upload`-like chart mode
3. New `chart_renderer.py` parses fallback tables and renders PNG files locally
4. `image_handler.py` uploads the rendered PNG to WeChat
5. The chart block is rewritten into WeChat-safe `<figure><img ...></figure>` or plain `<img>`
6. If rendering/upload fails, fall back to the existing table-preserving behavior

### Advantages

- Most compatible with the current report output
- Lowest runtime complexity among chart-image solutions
- No dependency on browser engines or Chart.js runtime timing
- Failure mode is simple and safe: keep table fallback
- Easy to reuse later for other channels that also prefer static images

### Weaknesses

- The visual result is a table-image, not the original chart visual
- Aesthetic quality depends on local table-to-image typography and layout quality
- Large tables may require paging, wrapping, or width compression

### Main Risks

- Chinese font availability and consistent rendering across environments
- Output image size must stay within WeChat upload limits
- Very wide tables may become unreadable if compressed into a single image

### Engineering Complexity

- Low to medium

### Recommended Ownership Split

- `image_handler.py`: strategy selection and upload coordination
- new `chart_renderer.py`: HTML table to PNG rendering
- `report_html_cleaner.py`: only final article node simplification

## Option 2: Reuse ChartToSVG Then Convert To PNG

### Idea

Reuse `radar_engines/ReportEngine/renderers/chart_to_svg.py` to convert chart payloads into static SVG, then convert SVG into PNG for WeChat upload and embed the resulting image into the final article body.

### Data Source

- Source: chart widget payload embedded in the report HTML
- Required input:
  - `canvas[data-config-id]`
  - corresponding JSON payload script generated by `html_renderer.py`

### Expected Pipeline

1. `report_html_cleaner.py` or a new chart adapter extracts chart payload JSON from report HTML
2. New adapter module maps report widget payload into `chart_to_svg.py` input format
3. `chart_to_svg.py` produces SVG
4. SVG is converted into PNG
5. `image_handler.py` uploads the PNG to WeChat
6. The original chart block is replaced with WeChat-safe `<img>` or `<figure>`
7. If conversion fails, degrade to table fallback instead of failing publish

### Advantages

- Much closer to the original chart appearance than table-image output
- Reuses upstream report engine capability instead of duplicating chart drawing logic
- No browser dependency
- Better long-term path if chart fidelity matters in multi-channel publishing

### Weaknesses

- Requires reliable extraction of chart payload JSON from report HTML
- Coupled more tightly to ReportEngine widget schema and rendering conventions
- SVG-to-PNG conversion adds another processing stage
- Some chart types or plugins may still render differently from front-end Chart.js output

### Main Risks

- Payload schema drift between ReportEngine HTML and WeChat adapter expectations
- `chart_to_svg.py` coverage may not perfectly match all widget types in production reports
- Font, theme, and color consistency may differ from HTML report output

### Engineering Complexity

- Medium

### Recommended Ownership Split

- new `chart_payload_adapter.py`: extract and normalize widget payloads from report HTML
- new `chart_renderer.py`: wrap `chart_to_svg.py` + SVG-to-PNG conversion
- `image_handler.py`: upload coordination and fallback routing

## Comparison

| Dimension | Fallback Table To PNG | ChartToSVG To PNG |
|---|---|---|
| Visual fidelity | Low to medium | Medium to high |
| Stability | High | Medium |
| Runtime dependency | Low | Medium |
| Current report compatibility | Very high | Medium to high |
| Implementation complexity | Low to medium | Medium |
| Failure recovery | Easy | Moderate |
| Coupling to ReportEngine internals | Low | Medium to high |

## Recommendation

### Phase 1 Recommendation

Implement **Fallback Table To PNG** first.

Reasoning:

- The latest real reports already contain complete `chart-fallback` blocks
- This gives the highest delivery certainty for WeChat draft publishing
- The user goal is currently ?drafts that are reviewable and not blank,? not perfect chart fidelity
- This path preserves a clean rollback to the existing `fallback_table` behavior

### Phase 2 Recommendation

Evaluate **ChartToSVG To PNG** as a fidelity upgrade layer.

This should be built only after Phase 1 is stable and should degrade back to fallback tables when SVG conversion fails.

## Suggested Rollout Order

1. Keep `fallback_table` as the default mode
2. Continue supporting plain `<img>` upload for reports that already include real images
3. Add `chart-fallback table -> PNG -> upload`
4. After that, add `chart_to_svg -> PNG -> upload` for selected chart types
5. Preserve table fallback as the final safety net at all times

## Architecture Constraint

The media solution should remain modular and should not move rendering logic into `service.py`.

Preferred module structure:

- `service.py`: orchestration only
- `report_html_cleaner.py`: article extraction and sanitization
- `image_handler.py`: media policy routing and upload coordination
- `chart_renderer.py`: static chart asset generation
- optional `chart_payload_adapter.py`: report chart payload extraction and normalization

## Final Decision Guidance

If the immediate priority is stable WeChat review drafts, choose:

- `chart-fallback table -> PNG`

If the later priority becomes chart visual fidelity, add:

- `chart_to_svg -> PNG`

The two strategies are not mutually exclusive. The correct long-term design is:

- stable table-image rendering as the first production layer
- SVG chart rendering as the later quality upgrade layer
