# 微信图表图片策略方案

## 背景

当前微信草稿发布流程在 `clawradar/publishers/wechat/` 中已将职责拆分如下：

- `service.py`：仅负责发布编排
- `report_html_cleaner.py`：报告 HTML 提取与微信安全简化
- `markdown_converter.py`：Markdown 转微信 HTML
- `image_handler.py`：媒体策略路由

目前渠道支持四种图片策略：

- `drop` — 丢弃
- `placeholder` — 占位符
- `fallback_table` — 表格回退
- `upload` — 上传

当前用于发布的最新真实报告中，实际结构分布为：

- `img = 0`
- `canvas = 8`
- `svg = 3`
- `chart-fallback = 8`

这说明当前行为：

- `upload` 模式对真实 `<img>` 节点已能正常工作
- 最新报告在微信中仍不显示图片，是因为报告中包含的是 chart canvas 而非内联 `<img>`

本文档故意排除浏览器截图路径，仅对两种实用的图表图片策略进行比较。

## 来自 ReportEngine 的上游线索

报告管线已经暴露了两个有用的属性：

1. `html_renderer.py` 将图表组件输出为 `canvas + chart-fallback`
2. `chart_to_svg.py` 已存在，能将 Chart.js 类组件载荷转换为静态 SVG

这意味着微信图表图片支持不需要从抓取运行中的浏览器开始。上游报告引擎已同时提供：

- 文本/表格回退表示
- 静态图表渲染方向

## 目标

在不依赖无头浏览器截图的前提下，为图表引入稳定的微信文章图片策略。

方案应满足：

- 保留当前安全的默认发布路径
- 避免将重渲染逻辑塞入 `service.py`
- 保持微信草稿工作流便于审查
- 与未来多渠道发布兼容

## 方案一：Chart 回退表格转 PNG

### 思路

从报告 HTML 中读取每个 `.chart-fallback table`，将表格渲染为本地静态 PNG，通过微信 `media/uploadimg` 上传 PNG，然后在最终微信文章正文中将 chart 块替换为 `<img>`。

### 数据来源

- 来源：`html_renderer.py` 输出的 HTML
- 所需输入：`.chart-fallback table`
- 不依赖前端 JavaScript 执行

### 预期流程

1. `report_html_cleaner.py` 按现有方式提取文章 HTML
2. `image_handler.py` 在 `upload` 类图表模式下检测 chart 容器
3. 新增 `chart_renderer.py` 解析回退表格并在本地渲染 PNG
4. `image_handler.py` 将渲染后的 PNG 上传至微信
5. Chart 块被改写为微信安全的 `<figure><img ...></figure>` 或纯 `<img>`
6. 渲染/上传失败时回退到现有的表格保留行为

### 优点

- 与当前报告输出兼容性最高
- 图表图片方案中运行时复杂度最低
- 不依赖浏览器引擎或 Chart.js 运行时时序
- 失败模式简单安全：保留表格回退
- 后续易于复用于其他同样偏好静态图片的渠道

### 缺点

- 视觉效果是表格图片，而非原始图表样式
- 美观程度取决于本地表格转图片的排版与字体质量
- 大表格可能需要分页、换行或宽度压缩

### 主要风险

- 中文字体在不同环境下的可用性与一致性
- 输出图片尺寸必须在微信上传限制内
- 过宽的表格压缩为单张图片后可能难以阅读

### 工程复杂度

- 低到中

### 推荐职责拆分

- `image_handler.py`：策略选择与上传协调
- 新增 `chart_renderer.py`：HTML 表格转 PNG 渲染
- `report_html_cleaner.py`：仅负责最终文章节点简化

## 方案二：复用 ChartToSVG 再转 PNG

### 思路

复用 `radar_engines/ReportEngine/renderers/chart_to_svg.py` 将 chart 载荷转换为静态 SVG，再将 SVG 转为 PNG 进行微信上传，最终图片嵌入文章正文。

### 数据来源

- 来源：嵌入报告 HTML 中的 chart widget 载荷
- 所需输入：
  - `canvas[data-config-id]`
  - `html_renderer.py` 生成的对应 JSON 载荷脚本

### 预期流程

1. `report_html_cleaner.py` 或新增 chart 适配器从报告 HTML 中提取 chart 载荷 JSON
2. 新增适配器模块将报告 widget 载荷映射为 `chart_to_svg.py` 输入格式
3. `chart_to_svg.py` 生成 SVG
4. SVG 转为 PNG
5. `image_handler.py` 将 PNG 上传至微信
6. 原始 chart 块替换为微信安全的 `<img>` 或 `<figure>`
7. 转换失败时降级为表格回退，而不是让发布失败

### 优点

- 视觉效果比表格图片方案更接近原始图表
- 复用上游报告引擎能力，避免重复造图表绘制轮子
- 无浏览器依赖
- 图表保真度在后续多渠道发布中价值更高

### 缺点

- 需要从报告 HTML 中可靠提取 chart 载荷 JSON
- 与 ReportEngine widget schema 和渲染约定的耦合更紧
- SVG 转 PNG 增加了一个处理阶段
- 某些图表类型或插件仍可能与前端 Chart.js 输出有差异

### 主要风险

- ReportEngine HTML 与微信适配器之间的载荷 schema 漂移
- `chart_to_svg.py` 覆盖面可能不完全匹配生产报告中的所有 widget 类型
- 字体、主题、颜色一致性可能与 HTML 报告输出不同

### 工程复杂度

- 中

### 推荐职责拆分

- 新增 `chart_payload_adapter.py`：从报告 HTML 提取并归一化 widget 载荷
- 新增 `chart_renderer.py`：封装 `chart_to_svg.py` + SVG 转 PNG
- `image_handler.py`：上传协调与回退路由

## 对比

| 维度 | 回退表格转 PNG | ChartToSVG 转 PNG |
|---|---|---|
| 视觉保真度 | 低到中 | 中到高 |
| 稳定性 | 高 | 中 |
| 运行时依赖 | 低 | 中 |
| 当前报告兼容性 | 非常高 | 中到高 |
| 实现复杂度 | 低到中 | 中 |
| 失败恢复 | 容易 | 中等 |
| 与 ReportEngine 内部耦合 | 低 | 中到高 |

## 推荐方案

### 第一阶段推荐

优先实现**回退表格转 PNG**。

理由：

- 最新真实报告中已包含完整的 `chart-fallback` 块
- 为微信草稿发布提供最高的交付确定性
- 当前用户目标是"可审查、不空白"的草稿，而非完美的图表保真度
- 此路径保留了干净回退到现有 `fallback_table` 行为的能力

### 第二阶段推荐

评估 **ChartToSVG 转 PNG** 作为保真度升级层。

仅在第一阶段稳定后再构建，且 SVG 转换失败时应回退到表格回退。

## 建议实施顺序

1. 保持 `fallback_table` 为默认模式
2. 继续支持对已包含真实图片的报告做纯 `<img>` 上传
3. 新增 `chart-fallback table -> PNG -> upload`
4. 之后对选定图表类型新增 `chart_to_svg -> PNG -> upload`
5. 始终保留表格回退作为最终兜底

## 架构约束

媒体方案应保持模块化，不应将渲染逻辑移入 `service.py`。

推荐模块结构：

- `service.py`：仅负责编排
- `report_html_cleaner.py`：文章提取与净化
- `image_handler.py`：媒体策略路由与上传协调
- `chart_renderer.py`：静态 chart 资源生成
- 可选 `chart_payload_adapter.py`：报告 chart 载荷提取与归一化

## 最终决策指引

如果当前优先级是稳定的微信审查草稿，选择：

- `chart-fallback table -> PNG`

如果后续优先级为图表视觉保真度，追加：

- `chart_to_svg -> PNG`

两种策略不是互斥的。正确的长期设计是：

- 稳定的表格图片渲染作为第一生产层
- SVG 图表渲染作为后续质量升级层
