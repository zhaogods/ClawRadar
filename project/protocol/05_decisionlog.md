# 决策记录

## 1. 主系统决策
决定继续由 OpenClaw 担任项目主系统。原因是项目的核心价值仍然在于跨步骤的调度、判断、评分、写作编排、审核交付与留档，而不是单点采集或单次写作。

## 2. 四引擎整体保留决策
决定当前阶段完整保留 `MindSpider`、`QueryEngine`、`MediaEngine`、`ReportEngine` 四个核心引擎，不做内部子功能裁剪。原因是主线虽然只直接调用了各引擎的核心入口，但这些入口仍依赖引擎内部其他模块和共享资源；在未完成最小依赖图梳理前继续删除内部模块，风险高于收益。

## 3. 旧平台外壳剥离决策
决定先删除旧平台外壳，而不是先改四引擎内部。原因是：
- `ForumEngine`、`InsightEngine`、`SingleEngineApp`、`SentimentAnalysisModel` 已不属于当前主线运行边界；
- 这些目录和旧入口脚本会持续干扰项目边界认知；
- 先做外壳剥离，可以让后续依赖评估建立在更清晰的目录结构之上。

## 4. 双输入模式统一主链路决策
决定无论输入来自真实来源还是用户主题，都必须汇入同一条后续主链路，而不建立两套平行评分、写作、交付协议。原因是项目真正需要统一的是选题、评分、写作、交付与留档，而不是让不同输入方式演化为不同系统。

## 5. `real_source` P0 多源轮转决策
决定把 `real_source` 从“单默认源 + 先到先得”调整为“多默认源 + 轮转合并”。原因是旧实现存在三个明显问题：
- 默认只抓 `weibo`，来源天然单一；
- 默认上限过小，不利于多源召回；
- 前序来源容易吃满 quota，导致后序来源失去机会。

该决策最终落地为：
- 默认源扩展到 `weibo`、`zhihu`、`36kr`；
- 默认上限提升到 `10`；
- 多来源候选按轮转方式合并。

## 6. `user_topic` P0 多 Provider 融合决策
决定把 `user_topic` 从“首个 provider 成功即返回”调整为“多 Provider 收集后统一融合”。原因是旧实现会让 `Tavily`、`Bocha`、`Anspire` 退化成优先级链，而不是召回增强链，直接造成结果单一、来源单薄、补证据不足。

该决策最终落地为：
- 先分别尝试已配置的 provider；
- 对各 provider 结果按 URL 去重；
- 统一轮转合并后再截断；
- `applied_source_ids` 记录全部实际参与 Provider。

## 7. `project/reports/` 本地化决策
决定 `project/reports/` 只作为本地分析目录存在，不进入远端仓库。原因是这些报告属于阶段性分析材料，而不是当前项目的正式可分发协议资产；把它们推送到远端会污染仓库主边界，也容易让本地分析材料被误当成正式交付物。

因此，后续执行规则明确为：
- 可在本地继续使用 `project/reports/`；
- 默认通过 [`.gitignore`](.gitignore) 忽略；
- 不再把该目录作为远端协议材料的一部分。

## 8. 下一阶段方向切换决策
决定下一阶段不再继续做大规模目录删除，而是转入“边界固化 + P1 增强”。原因是第一轮外壳剥离已经完成，继续删目录的收益快速下降，而当前更高价值的工作是：
- 确认四引擎最小运行依赖图；
- 提升 `real_source` / `user_topic` 的输入质量；
- 为后续精细瘦身建立事实依据。

## 9. P1 优先级决策
决定下一阶段优先处理输入层 P1，而不是优先处理 `ReportEngine` 内部瘦身。原因是：
- `real_source` / `user_topic` 直接影响候选质量与后续评分写作质量；
- 当前 P0 已经证明多源融合可稳定落地，继续做 P1 具有连续收益；
- `ReportEngine` 当前虽有兼容层残留，但仍处于“整体保留”边界内，先做依赖评估更合理。

## 10. 下一阶段完成标准决策
决定将以下内容作为下一阶段的完成标准：
- 形成四引擎最小运行边界清单；
- 完成 `real_source` / `user_topic` 的 P1 方案与至少一轮实现；
- 输出 `ReportEngine` 最小依赖评估结果；
- 明确哪些模块属于主线直接依赖、哪些属于间接依赖、哪些属于纯兼容保留。
