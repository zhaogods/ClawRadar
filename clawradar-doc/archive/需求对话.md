ClawRadar 这个 AI 助手项目，本质上不是“一个聊天机器人”，而是一个你自己托管的 AI 助手网关。你在本机或服务器上运行一个 `Gateway`，它把 Telegram、WhatsApp、Slack、Discord、Signal、iMessage、WebChat 等入口统一接到同一个 agent 上，所以你可以从不同渠道随时给自己的 AI 助手发消息。 [citation,docs.clawradar.ai](fda653) [citation,docs.clawradar.ai](3023dc)

它的核心设计是“Gateway + channels + agent + nodes”。Gateway 常驻在 `127.0.0.1:18789`，通过同一个 WebSocket/HTTP 入口服务 CLI、浏览器控制台、WebChat，以及 macOS/iOS/Android 节点；节点连上后还能给 agent 暴露设备能力，比如 `canvas`、`camera`、`screen.record`、`location.get`。 [citation,docs.clawradar.ai](3023dc) [citation,docs.clawradar.ai](e1f1fc)  
换句话说，ClawRadar 更像一个“AI 中枢”，而不是单独的一套前端应用。 [citation,docs.clawradar.ai](fda653)

上手门槛不算高。官方文档写的是推荐 `Node 24`，兼容 `Node 22.14+`，准备一个模型提供商的 API key 即可；安装后跑 `clawradar onboard --install-daemon`，再用 `clawradar gateway status` 和 `clawradar dashboard` 验证网关和控制台是否工作。官方宣称整个初始流程大约 5 分钟。 [citation,docs.clawradar.ai](22555f)  
默认控制台走浏览器，本地地址就是网关暴露的 UI；如果你想从手机上用，文档里也明确提到 Telegram 是比较快的入门渠道。 [citation,docs.clawradar.ai](22555f) [citation,docs.clawradar.ai](fda653)

它和很多“AI agent 项目”最大的区别，是它把多渠道接入、会话路由、工具调用和设备节点放进了同一套网关模型里。仓库 README 明确列了多渠道 inbox、多 agent 路由、浏览器控制、Live Canvas、技能系统、移动节点、Webhook/Cron 等能力。 [citation,github.com](19d452)  
所以如果你的目标是“把 AI 当个人常驻助手，从手机、桌面、聊天软件都能调”，ClawRadar 是对这个场景做得比较系统的一类项目。 [citation,github.com](19d452) [citation,docs.clawradar.ai](fda653)

安全模型上要注意一点：它默认是“个人助理信任模型”，也就是一个 Gateway 对应一个可信操作者边界，不是给互不信任的多人共用同一个 agent/gateway 设计的。官方文档明确说它不把单实例共享网关视为 hostile multi-tenant 安全边界。 [citation,docs.clawradar.ai](2241d6)  
默认思路是先做访问控制，再谈智能：谁能给 bot 发消息、它能在哪些群里响应、哪些工具能调用、是否放进沙箱。DM 推荐走 `pairing` 或 allowlist，群聊推荐强制 mention，高风险工具如 `exec`、`browser`、`web_search`、`web_fetch` 应限制给可信 agent，沙箱也是建议启用的。 [citation,docs.clawradar.ai](2241d6)

简化理解的话，可以把它看成：
1. 一个本地/自托管的 AI 助手后端。
2. 一个把多聊天渠道统一接进来的网关。
3. 一个支持工具、文件、浏览器、设备节点的 agent 运行环境。
4. 一个默认偏“个人使用”的系统，不适合直接拿来做多人对抗场景。 [citation,docs.clawradar.ai](fda653) [citation,docs.clawradar.ai](2241d6)

- `hot_topics.json`
- `today_news_digest.json`
- `candidate_topics.json`

但这个模式对你现在时间不友好，不建议作为首选。

**Agent 设计草案**
这版建议用 `4 个 agent`，但真正长期跑的主入口只有 `chief`。`chief` 是 ClawRadar 的主角，其他 agent 只是它调用的数字编辑部成员。ClawRadar 原生支持多 agent 路由、隔离 workspace、`sessions_spawn` 子代理、`cron` 定时任务和 `skills` 机制，这套设计是顺着它的能力来的。 [citation,docs.clawradar.ai](c8d9a3) [citation,docs.clawradar.ai](af91c5) [citation,docs.clawradar.ai](7399ff) [citation,docs.clawradar.ai](08d6fa)

1. `chief`
定位：总编排器。  
职责：被 `cron` 定时唤醒，调度其他 agent，落盘保存结果，最终推送到飞书。  
建议工具：
- `read`
- `write`
- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`
- `message`
- `session_status`

2. `radar`
定位：热点雷达研究员。  
职责：读取 BettaFish 产出的候选热点，做补充抓取、去重、聚类、时间线生成，返回候选主题包。  
建议工具：
- `web_search`
- `web_fetch`
- `exec`
- `session_status`
- 可选 `read`

3. `editor`
定位：选题编辑。  
职责：对 `radar` 返回的候选主题打分，给出栏目角度、为什么值得写、为什么不值得写。  
建议工具：
- 无工具或仅 `session_status`
- 不建议给 `exec`、`browser`

4. `writer`
定位：公众号作者。  
职责：根据 `editor` 选中的主题和证据包，生成标题、提纲、初稿、飞书摘要。  
建议工具：
- 无工具或仅 `session_status`
- 不建议给 `web_search`
- 不建议给 `exec`

**为什么这样分**
这套拆分能解决你现在的三个问题：
- `radar` 解决“热点不及时”
- `editor` 解决“选题不精准”
- `writer` 解决“内容不充实但又不能空泛”

同时它能避免“全能大模型一把梭”导致的泛化输出。`chief` 只负责编排和归档，内容判断分散到专门角色，更像一个真实编辑部。

**推荐的工具权限**
尽量让工具权限最小化。ClawRadar 的工具可以按 agent 做 allow/deny，工具组也支持 `group:web`、`group:fs`、`group:sessions`、`group:messaging` 等做限制。 [citation,docs.clawradar.ai](66bd96) [citation,docs.clawradar.ai](9bca52)

推荐原则：
- `chief` 只有文件、会话、消息能力，不做外部抓取。
- `radar` 才能访问网络和调用 BettaFish 脚本。
- `editor` 和 `writer` 尽量做“上下文内推理”，减少跑偏。

**Skill 设计草案**
Skills 放在 agent workspace 的 `skills/` 下，优先级最高；共享 skill 可以放到 `~/.clawradar/skills`。ClawRadar 文档明确支持按 agent 限定 skills 可见范围。 [citation,docs.clawradar.ai](08d6fa) [citation,docs.clawradar.ai](6ec843) [citation,docs.clawradar.ai](02e2c8)

我建议只做 `6 个 skill`，够比赛了。

1. `factory_orchestrator`
绑定 agent：`chief`  
作用：定义整条流水线顺序，约束 `chief` 不亲自写稿，只调度、保存、推送。  
核心规则：
- 先 `radar`，再 `editor`，再 `writer`
- 每轮只选 `1-2` 个主题进入写稿
- 所有中间结果必须保存为文件
- 未经 `editor` 评分的主题禁止进入 `writer`

2. `bettafish_signal_ingest`
绑定 agent：`radar`  
作用：规定如何读取或执行 BettaFish 输出，产出统一 `candidate_topics.json`。  
核心规则：
- BettaFish 只作为候选信号源
- 输出统一字段：`topic / keywords / source_urls / source_titles / source_count / time_range / summary`
- 如果候选主题超过 `10` 个，只保留最有代表性的 `8` 个

3. `timeline_cluster`
绑定 agent：`radar`  
作用：把多篇来源整合成一个事件时间线。  
核心规则：
- 同公司同事件聚成一组
- 时间线至少包含 `3` 个节点
- 每个节点必须带来源
- 如果证据不足，不生成“结论”，只输出“待确认”

4. `topic_scorecard`
绑定 agent：`editor`  
作用：对主题做评分。  
评分维度建议固定成：
- `时效性`
- `传播潜力`
- `信息密度`
- `公众号受众匹配度`
- `新鲜度`
每项 `1-10` 分，并输出：
- 总分
- 值不值得写
- 推荐理由
- 风险提示

5. `angle_planner`
绑定 agent：`editor`  
作用：防止选题太泛。  
核心规则：
- 每个主题至少给 `3` 个具体角度
- 每个角度都要注明“适合谁看”
- 每个角度都要给一句“为什么这个角度不空泛”

6. `wechat_article_factory`
绑定 agent：`writer`  
作用：从证据包生成公众号材料。  
输出顺序固定：
- 标题 3 个
- 导语 1 段
- 提纲 1 份
- 初稿 1 篇
- 飞书摘要 1 条

**最关键的 skill 约束**
你现在的问题不是不会写，而是容易写“空”。所以 `wechat_article_factory` 要加硬约束：

- 必须引用 `timeline` 中的事实点
- 必须写“这件事为什么重要”
- 必须写“目前还不确定什么”
- 没有证据的判断用“可能/或许/待确认”
- 禁止泛泛而谈的句式，比如“这一事件引发行业广泛关注”这种空话，除非后面有证据支撑

```md
# TITLES
1. OpenAI 又更新了，但真正值得开发者看的不是“新模型”三个字
2. 一次发布，两个信号：OpenAI 这轮模型与 API 更新到底在指向什么
3. 从 API 到工作流：为什么这次 OpenAI 更新比普通产品发布更值得写

# LEAD
OpenAI 再次发布新模型与 API 更新，这类新闻表面上看已经越来越常见，但真正值得注意的，不是“又上新了”，而是它对开发者工作流和产品判断释放出的新信号。从官方公告、开发者文档更新到多家技术媒体跟进，这次更新至少说明一件事：模型竞争正在继续，但真正被放大的，是开发者接口和应用层节奏。

# OUTLINE
1. 这次更新到底发生了什么
2. 为什么开发者比普通读者更应该关心这次发布
3. API 与工作流层面的变化意味着什么
4. 这次更新背后释放出的平台节奏信号
5. 目前还不能下结论的地方
6. 编辑判断：为什么这条值得今天写

# DRAFT
如果只把这次 OpenAI 的动作理解成“又发了一个新模型”，那大概率会低估它的内容价值。真正值得写的部分，不在发布动作本身，而在它如何影响开发者、产品团队以及围绕 API 进行构建的人。

从目前公开信息看，这次更新至少包含两个层面。第一是模型层面的迭代。无论是新能力释放、可用性变化，还是对既有能力的重新包装，模型更新本身依然是最容易被传播的部分。第二是接口与工作流层面的变化。相比前者，这部分更少被标题突出，但往往更直接地影响开发者的调用方式、产品团队的接入判断，以及企业在内部评估模型能力时的成本与节奏。

这也是为什么这条新闻值得从开发者 API 视角来写。对于真正使用大模型的人来说，模型发布从来不是抽象新闻，而是具体的工作流问题：接口有没有变化，文档有没有更新，旧的接入逻辑要不要改，新的能力是否值得马上试，产品路线是否需要重新比较。一个模型更新如果没有进入实际调用和应用流程，那它对行业的影响往往会被高估；反过来，如果一次看似平常的更新改变了开发者体验和接入效率，它的真实价值反而可能被低估。

从媒体跟进方式也能看出这一点。早期报道通常集中在“发布了什么”，而后续的开发者讨论更关心“这能不能真正用起来”。这两个层次之间的落差，恰恰是公众号选题最值得抓的地方。对普通读者来说，看到的是一条产品新闻；对开发者和产品团队来说，看到的是平台节奏、接口能力和生态优先级的再排序。
```

**10. 角色设计**

**12. 评分机制设计**
- 评分维度固定为 5 个：`timeliness`、`virality`、`information_density`、`audience_fit`、`novelty`。
- 每个维度范围 `1-10`，总分 `0-50`。
- 推荐阈值：
  - `41-50`：`write`
  - `31-40`：`watch` 或有条件 `write`
  - `0-30`：`skip`
- 每个主题必须输出：
  - `score_total`
  - `decision`
  - `why_write`
  - `risk_notes`
  - `angles`
- 原则：
  - 优先公司级具体事件，而不是泛行业趋势。
  - 优先来源扎实、信息密度高的主题。
  - 若热点已被高度重复覆盖且无新信息，降低 `novelty`。
  - 若来源少、事实弱、时间线不完整，即使传播性高也应降分。

BettaFish 本身是一套完整的多智能体舆情分析系统，包含 `QueryEngine`、`MediaEngine`、`InsightEngine`、`ReportEngine`，以及负责抓取与话题提取的 `MindSpider`；其中 `MindSpider` 又分成 `BroadTopicExtraction` 和 `DeepSentimentCrawling` 两步。 [citation,github.com](bdcaf7) [citation,github.com](9fd4f2)

**核心原则**
你这次二开的目标不是“做一个 BettaFish + ClawRadar 的双主脑系统”，而是：

- `BettaFish` 负责：找到“今天可能值得写的事”
- `ClawRadar` 负责：判断“今天真正该写什么、怎么写、发给谁”

这条边界不能动。  
一旦 BettaFish 开始负责完整分析、报告生成、内容编排，它就会抢掉 ClawRadar 的主角位置。

**推荐二开层级**
我建议你做 `L2`，不要做 `L3`。

`L1：最小接入`
- 不改 BettaFish 核心逻辑
- 只读取它现有输出
- 外部再做一个转换脚本

优点：快  
缺点：输出格式不可控，后面 `editor` 和 `writer` 容易吃脏数据

`L2：推荐方案`
- 改 `BroadTopicExtraction`
- 加一个 `ClawRadar Exporter`
- 加一个轻量过滤与去重层
- 统一输出 `candidate_topics.json`

优点：最适合比赛，边界清楚，叙事清晰  
缺点：需要轻量改造

`L3：不推荐本轮做`
- 深改 `DeepSentimentCrawling`
- 接社媒深爬
- 接情感分析
- 接论坛协作
- 接完整 ReportEngine

优点：上限高  
缺点：复杂、抢戏、容易翻车

**建议保留与舍弃的模块**
建议保留：
- `MindSpider/BroadTopicExtraction/get_today_news.py`
- `MindSpider/BroadTopicExtraction/topic_extractor.py`
- 可选 `QueryEngine` 的新闻广搜能力

建议舍弃：
- `DeepSentimentCrawling`
- `MediaEngine`
- `InsightEngine`
- `ReportEngine`
- `ForumEngine`

原因很简单。  
你现在的目标不是“舆情分析平台”，而是“内容选题雷达”。社媒深爬、情感分析、论坛协作、完整报告生成都会把系统复杂度和叙事重心拉走。BettaFish 自己的 README 也说明它的目标是舆情分析与报告闭环，而不是内容编辑工作流。 [citation,github.com](bdcaf7)

**最合理的二开产物**
你应该把 BettaFish 最终改造成只输出一类文件：

```json
candidate_topics.json
```

这个文件就是给 ClawRadar 吃的“候选热点池”，不要让 BettaFish 直接产出长报告。

推荐输出字段：

```json
{
  "generated_at": "ISO-8601",
  "input_mode": "bettafish_export",
  "upstream": {
    "source": "BettaFish",
    "run_id": "string",
    "freshness_note": "string"
  },
  "topics": [
    {
      "topic_id": "string",
      "topic": "string",
      "keywords": ["string"],
      "event_type": "model_release|product_launch|funding|earnings|org_change|policy|open_source_release|infra|chip|partnership|other",
      "company": "string",
      "source_titles": ["string"],
      "source_urls": ["string"],
      "source_count": 0,
      "time_range": {
        "start": "ISO-8601 or null",
        "end": "ISO-8601 or null"
      },
      "summary": "string",
      "signal_strength": "high|medium|low",
      "freshness_status": "fresh|recent|aging|stale",
      "notes": ["string"]
    }

```

这个 schema 的目的不是“把 BettaFish 的全部能力输出出来”，而是“让 ClawRadar 后续能稳定处理”。

**具体怎么改**
下面是最实用的改造方案。

**第一步：fork 一个比赛专用分支**
不要直接在主分支乱改。建议新开分支：

- `feature/clawradar-signal-layer`

你在文章里也更容易说清楚：
- 原项目是什么
- 我改了什么
- 我只拿了哪一层能力

**第二步：缩窄信源**
BettaFish 原本偏舆情和社媒，覆盖面很广。 [citation,github.com](bdcaf7) [citation,github.com](74b502)  
你现在要的是“AI/科技公司编辑信号”，所以第一刀应该砍信源范围。

建议只保留这几类来源：
- 公司官方博客 / newsroom
- 官方产品发布页 / changelog
- 官方开发者文档更新
- GitHub release / repo update
- 主流科技媒体
- 商业媒体里与 AI 公司、财报、并购、组织变动相关的栏目
- 政策和监管公开源

不建议这轮主打：
- 登录后才能稳定抓的社媒平台
- 评论区与情绪数据
- 大量 UGC

建议在 `MindSpider/BroadTopicExtraction/get_today_news.py` 附近做一个“比赛模式”来源白名单，或者新增一个单独配置文件，比如：

```text
MindSpider/config/clawradar_sources.yaml
```

内容思路：
- `official_sources`
- `tech_media_sources`
- `finance_sources`
- `policy_sources`

这样你后面解释系统时，会很像一个“编辑信号源白名单”，而不是一个泛爬虫。

**第三步：改造话题提取逻辑**
`topic_extractor.py` 现在更偏“热点话题提取”，你要把它改成“编辑候选主题提取”。 [citation,github.com](9fd4f2)  
核心变化不是让它提更多话题，而是让它提更适合写稿的话题。

建议让它多输出这些字段：
- `company`
- `event_type`
- `topic`
- `keywords`
- `why_now`
- `signal_strength`
- `source_count`

你要把 prompt 从“今天网上大家在讨论什么”改成“今天有哪些 AI/科技事件足够具体、足够新、足够有信息密度，能支撑一篇公众号选题”。

一句话总结：
- 从 `舆情热点抽取`
- 改成 `编辑候选主题抽取`

**第四步：新增一个 ClawRadar 导出器**
这是最重要的一步。  
建议你不要改得太散，而是集中新增一个专门的导出模块，比如：

```text
MindSpider/exports/clawradar_exporter.py
```

这个脚本只做四件事：
- 读取当轮抓取结果
- 做轻量过滤
- 做轻量去重
- 输出 `candidate_topics.json`

它的输入可以是：
- `BroadTopicExtraction` 直接产出的主题
- 或者落库后的 `daily_news / daily_topics` 数据

README 已经提到这块有 `daily_news`、`daily_topics`、`topic_news_relation` 等表。 [citation,github.com](9fd4f2)

Exporter 的职责一定要窄：
- 不是报告生成器
- 不是最终分析器
- 不是编辑评分器

它只是“清洗后的热点候选池生产器”。

**第五步：新增编辑过滤器**
建议加一个轻量过滤文件，比如：

```text
MindSpider/filters/clawradar_editorial_filter.py
```

这里做 3 类判断：

- 过滤太泛的话题
  - 例如“AI 行业持续升温”
  - 例如“开源 Agent 趋势继续发展”
- 过滤证据太弱的话题
  - 来源太少
  - 没有明确 trigger event
- 过滤与公众号受众不匹配的话题
  - 话题很热，但与你目标栏目无关

这一层非常关键，因为它会直接提升 ClawRadar 后面的写作质量。

**第六步：新增轻量去重和聚合**
建议加一个模块，比如：

```text
MindSpider/postprocess/clawradar_topic_merge.py
```

去重规则建议用：
- `company + event_type + time_window`
- `topic text semantic similarity`
- `source overlap`

举例：
- “OpenAI 发布新模型”
- “OpenAI 更新 API”
- “OpenAI 推出开发者新能力”

如果是同一轮发布，你应该合并成同一主题，而不是让 ClawRadar 后面拿到 3 个几乎一样的候选题。

**第七步：把 BettaFish 改成无界面模式**
比赛里不要展示 BettaFish 的完整 UI，也不要展示它的完整报告引擎。  
你应该把它改造成一个“后台信号层”。

推荐运行方式：
- BettaFish 在后台定时跑
- 只生成 `candidate_topics.json`
- 写入 ClawRadar workspace

推荐输出路径：

```text
~/.clawradar/workspace-chief/reports/YYYY-MM-DD/candidate_topics.json
```

这样 ClawRadar 可以直接 `read` 读取，不需要复杂接口对接。

ClawRadar 的 workspace 本身就是为这种“长期上下文 + 文件沉淀”准备的。 [citation,docs.clawradar.ai](6ec843)

**第八步：用 ClawRadar 统一调度**
这里是重点。  
即使 BettaFish 也能定时跑，你比赛里最好不要把“调度权”放在 BettaFish。

更好的方式是：
- ClawRadar `cron` 定时触发
- `radar` agent 用 `exec` 调用 BettaFish exporter
- exporter 写出 `candidate_topics.json`
- `radar` 再读这个文件，继续做时间线与事实补充

这样一来，整个运行链路就变成了：

```text
ClawRadar cron
-> ClawRadar radar agent
-> exec BettaFish exporter
-> candidate_topics.json
-> ClawRadar editor
-> ClawRadar writer
-> Feishu
```

评委一看就知道：
- ClawRadar 在编排
- BettaFish 只是工具化输入层

**1. `clawradar_exporter.py` 详细设计**
推荐你新增：

```text
MindSpider/exports/clawradar_exporter.py
```

它只做 5 件事：
- 读取 BettaFish 产生的候选新闻和主题
- 过滤掉不适合公众号选题的内容
- 合并重复事件
- 规范化成统一 schema
- 写出 `candidate_topics.json`

它不要做：
- 深度舆情报告
- 观点分析
- 情感分析汇总
- 最终选题评分
- 公众号写作

**输入来源**
优先级建议如下：
- `daily_topics`
- `daily_news`
- `topic_news_relation`
- 如果直接读表太重，可以先读 BettaFish 当前阶段的中间 JSON / Markdown 输出，再做统一转换。MindSpider README 已说明它会围绕 `daily_news`、`daily_topics`、`topic_news_relation` 等结构组织数据。 [citation,github.com](9fd4f2)

**输出原则**
- 输出越小越好
- 字段越稳定越好
- 必须为编辑判断服务
- 必须方便 ClawRadar 后续 `radar -> editor -> writer`

**推荐输出字段**
推荐最终格式如下：

```json
{

  "topics": [

      "topic_id": "20260407-openai-model-release",

```

**字段解释**
- `topic_id`
  用于跨天去重，建议用 `date + company + event_type + hash(title)`。
- `topic`
  中文可读名称，给 ClawRadar 后续直接使用。
- `canonical_topic`
  英文或标准化内部名，方便后续聚类与历史归档。
- `company`
  如果是公司事件，尽量提取出来。
- `event_type`
  强烈建议固定枚举值，不要自由文本。
- `trigger_event`
  说明为什么这个主题今天成立。
- `signal_strength`
  只是轻量信号，不是最终评分。
- `editorial_flags`
  这是给 ClawRadar `editor` 的捷径，比如 `official_source_present`、`pricing_change`、`developer_impact` 这种。
- `editor_hints`
  用来减少 `editor` 和 `writer` 的泛化倾向。
- `evidence_items`
  比单纯的 `source_urls` 更好，因为它已经把来源类型和片段带出来了。

**导出器处理流程**
建议在 `clawradar_exporter.py` 里拆成这些函数：

```python
def load_raw_topics(...)
def normalize_topic(...)
def detect_company(...)
def classify_event_type(...)
def dedupe_topics(...)
def compute_freshness(...)
def compute_signal_strength(...)
def build_editorial_flags(...)
def build_editor_hints(...)
def export_candidate_topics(...)
```

**最重要的三段逻辑**
第一段：`normalize_topic`
- 把标题去噪
- 去掉“传闻”“网友热议”“再度引发关注”这种空壳词
- 统一成“公司 + 事件”的格式

第二段：`dedupe_topics`
- 同一公司、同一事件类型、时间窗口接近，直接合并
- 不要把一次发布拆成 3 个相似主题

第三段：`build_editorial_flags`
- 把“这个主题为什么适合写”先编码出来
- 后面 ClawRadar `editor` 再做最终决策

**最小可运行导出器思路**
如果你现在不想深读 BettaFish 数据库表，先做个简单版：
- 输入：BettaFish 已经产出的新闻标题列表
- 输出：`candidate_topics.json`

也就是先跑“标题级导出器”，后面再升级成“数据库级导出器”。

**短版**
> 为了解决热点发现不及时的问题，我没有从零重写爬虫，而是对开源项目 BettaFish 做了轻量级二次开发。  
> 我只保留了其 `MindSpider/BroadTopicExtraction` 的新闻抓取与话题提取能力，并新增了一个面向 ClawRadar 的导出层 `clawradar_exporter`，将原本偏舆情分析的输出，收敛为一个专门服务内容选题的 `candidate_topics.json`。

**中版**
> 原始 BettaFish 是一套完整的多智能体舆情分析系统，目标是从新闻与社媒中挖掘舆情、做深度分析并生成报告。这个能力很强，但如果直接整套接入我的比赛作品，会出现两个问题：一是系统过重，二是会让 BettaFish 抢走 ClawRadar 的主角位置。  
> 因此，我采用了“削薄而不是增强”的二次开发策略：只保留 BettaFish 的热点发现层，对其做三项改造。  
> 第一，缩窄信源范围，只保留 AI 公司、科技媒体、财报、政策、开源发布等高信号来源；  
> 第二，增加编辑过滤器，过滤掉泛话题、弱来源和与目标公众号受众不匹配的内容；  
> 第三，新增 `clawradar_exporter.py`，将输出统一转成 `candidate_topics.json`，作为 ClawRadar 数字编辑部的输入。  
> 这样一来，BettaFish 只负责“发现值得关注的候选事件”，而 ClawRadar 才真正承担起工作流编排、时间线生成、选题打分、内容生成与飞书推送的核心角色。

**强调主从关系版**
> 这次我刻意没有把 BettaFish 当成主系统，而是把它改造成一个“上游热点信号层”。  
> 它只做一件事：给 ClawRadar 提供结构化的候选主题。  
> 所有真正体现产品价值的环节，包括事件聚类、多来源补充抓取、选题评分、最佳角度选择、公众号标题/提纲/初稿生成，以及最终的消息交付，都由 ClawRadar 完成。  
> 换句话说，BettaFish 负责喂信号，ClawRadar 负责养成内容。
