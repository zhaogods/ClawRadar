# 统一入口使用说明

本文只回答一件事：当前仓库里的统一入口如何调用、最小要传什么、分阶段模式如何选、以及 skill 应如何最小化组织调用。

所有说明都以当前真实实现为准，统一入口是 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)。本文不会把 [`project_protocol/stage10/`](project_protocol/stage10/) 历史材料改写成本轮 skill 方案证明，也不会把未实现事项写成已完成事实。

## 1. 统一入口定位

当前项目真实统一入口就是 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)。

推荐直接导入：

```python
from clawradar.orchestrator import topic_radar_orchestrate
```

这个入口负责把请求统一收口到同一条主链路，而不是让调用方自己拼装平行编排。当前主链路可概括为：

1. crawl：根据输入准备候选材料，并生成 `crawl_results`
2. topics：生成显式选题卡，并输出 `topic_cards`
3. score：评分并输出 `scored_events` / `score_results`
4. write：对可继续写作的事件生成 `content_bundles`
5. deliver：交付或留档，并输出 `delivery_receipt` / `delivery_result`
6. summarize：通过 `run_summary`、`stage_statuses`、`event_statuses` 收口结果

重要边界：

- 面向正式交付时，仓库根目录的 [`run_openclaw_deliverable.py`](run_openclaw_deliverable.py) 是 ClawRadar 的唯一推荐 launcher；
- 真正统一入口仍然是 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)；
- launcher、skill、直接 Python 调用都应收口到同一套 `entry_options` 协议；
- 不应再新增第二套平行顶层入口。

## 2. 最小调用方式

如果只是想把入口跑起来，最稳妥的最小调用方式仍然是直接调用 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)。

### 2.1 最小 Python 示例

```python
from clawradar.orchestrator import topic_radar_orchestrate

payload = {
    "request_id": "req-minimal-001",
    "trigger_source": "manual",
    "topic_candidates": [
        {
            "event_id": "evt-minimal-001",
            "event_title": "OpenAI 发布新模型能力更新",
            "event_time": "2026-04-09T08:00:00Z",
            "source_url": "https://example.com/openai-model-update"
        }
    ]
}

result = topic_radar_orchestrate(payload)
print(result["run_status"], result["final_stage"], result["decision_status"])
```

### 2.2 正式 launcher 示例

正式交付场景优先直接使用根目录 launcher：

```bash
python run_openclaw_deliverable.py --input-mode real_source --source-ids weibo --limit 5
python run_openclaw_deliverable.py --input-mode user_topic --topic "AI 智能体安全治理" --company "OpenAI" --track "企业智能体" --keywords 治理 审计
```

这个 launcher 会默认收口到：

- `entry_options.write.executor="external_writer"`
- `entry_options.delivery.target_mode="archive_only"`
- `entry_options.delivery.target="archive://openclaw_p0"`
- `entry_options.degrade.*="fail"`

运行完成后，优先看返回值里的 `output_root`。新版本会把一次运行的所有正式输出集中到同一个目录下，例如：

- `meta/`：`run_summary.json`、`entry_resolution.json`、`artifact_summary.json`、`errors.json`
- `stages/crawl/`、`stages/topics/`、`stages/score/`、`stages/write/`、`stages/deliver/`
- `reports/final/`、`reports/ir/`、`reports/chapters/`、`reports/logs/`
- `events/<event_id>/deliver/<timestamp>/scorecard.json`、`payload_snapshot.json`、`feishu_message.json`

`--runs-root` 现在表示“总输出根目录”；若不显式传入，默认使用项目根目录下的 `outputs/`。真正单次运行目录会自动生成为 `<runs_root>/<request_id>/<run_slug>/`。

最小事件字段仍建议至少提供：

- `event_id`
- `event_title`
- `event_time`
- `source_url`

这类最小输入通常只能稳定跑到 score；如果证据不足，结果会停在 `need_more_evidence`，不会自动进入写作和交付。这属于当前实现的正常行为。

## 3. skill 调用时的最小推荐方式

如果调用方来自 OpenClaw / Roo 侧，推荐让 skill 直接调用统一入口；面向正式交付时，优先复用根目录 `run_openclaw_deliverable.py`，不要再自行扩出第二套入口。

推荐做法：

1. skill 识别输入属于 `real_source`、`user_topic` 或已有 `inline_topic_cards`；
2. skill 只补齐最小必要字段；
3. skill 直接调用 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398)；
4. skill 回报 `run_status`、`final_stage`、`decision_status`、`entry_resolution` 与 `run_summary`。

不推荐做法：

- 先新增 `launcher.py`、`run_skill.py`、`main.py` 一类包装脚本；
- 绕开 [`topic_radar_orchestrate()`](clawradar/orchestrator.py:1398) 直接分别调用 crawl / topics / score / write / deliver；
- 把 skill 写成新的顶层编排系统。

## 5. `entry_options.input` 输入模式说明

统一入口的输入模式收口在 [`_build_entry_resolution()`](clawradar/orchestrator.py:236) 中。当前对外最值得关注的是以下模式。

### 5.1 `real_source`

当你希望先从真实来源拉取候选材料，再继续统一主链路时，使用 `real_source`。

```json
{
  "entry_options": {
    "input": {
      "mode": "real_source"
    }
  }
}
```

说明：

- `real_source` 不是“保证联网成功”的承诺；
- 成功时会在结果里记录 `entry_resolution.input.real_source_loaded=true`；
- 失败时是否回退，取决于 `entry_options.degrade.input_unavailable`；
- 这一模式通常用于来源扫描、热点采集或自动发现候选。

### 5.2 `user_topic`

当用户只给了主题、公司、关键词、赛道或意图，而没有现成候选事件时，使用 `user_topic`。

```json
{
  "user_topic": {
    "topic": "OpenAI 企业级智能体平台",
    "company": "OpenAI",
    "keywords": ["智能体", "企业服务"]
  },
  "entry_options": {
    "input": {
      "mode": "user_topic"
    }
  }
}
```

说明：

- `user_topic` 会先被 [`load_user_topic_payload()`](clawradar/topics.py:196) 规范化，再委托真实抓取层获取候选，再进入统一主链路；
- 正式语义是“围绕主题触发真实抓取”，而不是本地伪造 `user_topic://...` 候选；
- 成功时会在结果里记录 `entry_resolution.input.user_topic_loaded=true`、`user_topic_provider` 与 `user_topic_candidate_count`。

### 5.3 `inline_topic_cards`

当上游已经有显式选题卡，希望直接从 topics / score 之后继续时，使用 `inline_topic_cards`。

```json
{
  "topic_cards": [
    {
      "event_id": "evt-demo-001",
      "topic_title": "OpenAI 企业级智能体平台",
      "summary": "围绕企业级智能体协作与审计能力形成的选题卡。"
    }
  ],
  "entry_options": {
    "input": {
      "mode": "inline_topic_cards"
    }
  }
}
```

说明：

- 适合复用已有选题工件；
- 适合与 `score_only`、`write_only`、`resume` 配合使用；
- 如果未提供 `topic_cards`，统一入口会直接返回明确错误，而不是隐式猜测。

### 5.4 兼容输入：`inline_candidates` 与 `inline_normalized`

除了本轮重点模式之外，当前仍兼容：

- `inline_candidates`：直接使用 `topic_candidates`
- `inline_normalized`：直接使用 `normalized_events`

这两种模式主要用于兼容已有调用方与中间工件回放。

## 6. `entry_options.write` 与 `entry_options.delivery`

### 6.1 `entry_options.write`

常用字段：

- `enabled`
- `executor`：`openclaw_builtin` 或 `external_writer`
- `operation`：`generate`、`rewrite`、`regenerate_summary`

最常见写法：

```json
{
  "entry_options": {
    "write": {
      "enabled": true,
      "executor": "external_writer",
      "operation": "generate"
    }
  }
}
```

说明：

- 正式默认写作执行器已经切到 `external_writer`；
- `openclaw_builtin` 仍可显式指定，或通过 `write_unavailable=fallback_openclaw_builtin` 作为降级路径；
- `rewrite` 与 `regenerate_summary` 需要已有 `content_bundle`，其输入约束见 [`validate_write_payload()`](clawradar/writing.py:121)；
- 即使 `execution_mode="full_pipeline"`，如果 `enabled=false`，也会在 score 后直接收口。

### 6.2 `entry_options.delivery`

常用字段：

- `enabled`
- `target_mode`：`feishu` 或 `archive_only`
- `channel`
- `target`
- `time`

最推荐的本地调用方式：

```json
{
  "entry_options": {
    "delivery": {
      "target_mode": "archive_only",
      "target": "archive://openclaw-p0-local"
    }
  }
}
```

说明：

- 当前真实外部交付语义仍是 `feishu`；
- 本地验证优先推荐 `archive_only`；
- `archive_only` 由 [`build_archive_only_delivery_result()`](clawradar/delivery.py:497) 直接处理；
- 使用 `archive_only` 不代表跳过交付结果，只是避免真实发送。

## 7. `entry_options.degrade` 降级说明

`entry_options.degrade` 表示“请求能力不可用时怎么收口”，不是重试器。

当前常用选项：

- `input_unavailable`：`fail` / `fallback_inline_candidates` / `fallback_inline_normalized`
- `write_unavailable`：`fail` / `fallback_openclaw_builtin` / `skip`
- `delivery_unavailable`：`fail` / `archive_only`

示例：

```json
{
  "entry_options": {
    "input": {
      "mode": "real_source"
    },
    "write": {
      "executor": "external_writer"
    },
    "delivery": {
      "target_mode": "feishu",
      "channel": "feishu",
      "target": "feishu://openclaw/p0-review"
    },
    "degrade": {
      "input_unavailable": "fallback_inline_candidates",
      "write_unavailable": "fallback_openclaw_builtin",
      "delivery_unavailable": "archive_only"
    }
  }
}
```

所有 fallback 都会被记录到 `entry_resolution.degrade.fallbacks`，不会静默发生。

## 8. 执行模式最小说明

统一入口支持的执行模式定义在 [`OrchestratorExecutionMode`](clawradar/orchestrator.py:31)。当前最小说明如下。

### 8.1 `full_pipeline`

默认值。按当前输入尽可能继续往下执行，直到：

- 证据不足被收口；
- 写作被关闭；
- 交付被关闭；
- 或成功走到 deliver。

### 8.2 `crawl_only`

只准备并输出 `crawl_results`，不继续评分或写作。

适合：

- 只想先看采集结果；
- 想核对来源输入是否可用；
- 想把 crawl 工件留给下一步处理。

### 8.3 `topics_only`

执行到显式选题层并输出 `topic_cards` 后收口。

适合：

- 只想先看选题卡；
- 想先做人审或编辑筛选；
- 想把 `topic_cards` 作为后续 `score_only` / `resume` 输入。

### 8.4 `score_only`

只跑到 score，并返回 `scored_events` / `score_results`。

适合：

- 只想判断值不值得写；
- 想把入口当作批量选题评分器；
- 想在写作前先做人审。

### 8.5 `write_only`

基于已有 `scored_events` 组织写作，并在 write 完成后收口，不自动继续交付。

适合：

- 评分结果已经存在；
- 只想生成内容包；
- 想把交付留到后续单独执行。

### 8.6 `deliver_only`

基于已有 `content_bundles` 只执行交付。

适合：

- 内容包已经准备好；
- 需要重新投递或重新留档；
- 不希望重复前面的 crawl / topics / score / write。

### 8.7 `resume`

根据已有中间工件自动决定从哪一阶段继续。

当前已成立的恢复语义是：

- 有 `content_bundles` 时优先继续到 deliver；
- 有 `scored_events` 时优先继续到 write；
- 有 `topic_cards` 或 `normalized_events` 时优先继续到 score；
- 其他情况回到主流水线起点。

## 9. skill 取参最小化原则

当信息不足时，推荐按以下原则最小化向用户取参。

### 9.1 能默认就默认

skill 或 launcher 可优先自动补齐：

- `trigger_source=manual`
- `execution_mode=full_pipeline`
- `entry_options.write.executor=external_writer`
- `entry_options.delivery.target_mode=archive_only`
- `entry_options.delivery.target=archive://openclaw_p0`

### 9.2 只问真正缺失的业务输入

优先只追问以下几类：

1. 你要用哪一种输入：`real_source`、`user_topic`、还是已有 `inline_topic_cards`；
2. 如果是 `user_topic`，至少要给主题、公司、关键词中的有效信息；
3. 如果是 `deliver_only`、`write_only` 或 `resume`，需要说明已有哪类中间工件；
4. 只有当用户明确要求真实交付时，才继续追问交付目标。

### 9.3 推荐追问顺序

1. 你是按 `user_topic`、`real_source`，还是已有 `topic_cards` 来跑？
2. 默认按 `full_pipeline + archive_only` 执行，除非你明确指定 `crawl_only`、`topics_only`、`score_only`、`write_only`、`deliver_only`、`resume`。
3. 只有你明确要求真实发送时，才继续问 `channel` 与 `target`。

## 10. 返回结果里最关键的字段

最值得先看的字段是：

- `run_status`
- `final_stage`
- `decision_status`
- `entry_resolution`
- `stage_statuses`
- `event_statuses`
- `run_summary`

按阶段常看的工件：

- crawl：`crawl_results`
- topics：`topic_cards`
- score：`scored_events`、`score_results`
- write：`content_bundles`
- deliver：`delivery_receipt`、`delivery_result`

如果是正式 launcher 产物，建议直接从 `output_root` 查看：

- `meta/run_summary.json`
- `stages/score/scored_events.json`
- `stages/write/content_bundles.json`
- `stages/deliver/delivery_receipt.json`
- `events/<event_id>/deliver/<timestamp>/scorecard.json`

如果要把结果继续喂给下一步，优先复用这些结构化工件，而不是从日志里抠文本。

## 11. 本地验证建议

本地正式交付最推荐的调用组合是：

- `full_pipeline`
- `archive_only`
- `external_writer`

示例：

```python
import json
from pathlib import Path
from clawradar.orchestrator import topic_radar_orchestrate

base = Path("tests/fixtures/openclaw_p0_score_publish_ready_input.json")
payload = json.loads(base.read_text(encoding="utf-8"))
payload["request_id"] = "req-openclaw-local-demo"
payload["entry_options"] = {
    "write": {
        "executor": "external_writer"
    },
    "delivery": {
        "target_mode": "archive_only",
        "target": "archive://openclaw_p0"
    },
    "degrade": {
        "input_unavailable": "fail",
        "write_unavailable": "fail",
        "delivery_unavailable": "fail"
    }
}

result = topic_radar_orchestrate(payload)
print(json.dumps({
    "run_status": result.get("run_status"),
    "final_stage": result.get("final_stage"),
    "decision_status": result.get("decision_status"),
    "entry_resolution": result.get("entry_resolution"),
    "run_summary": result.get("run_summary")
}, ensure_ascii=False, indent=2))
```

若只做本地演示或故障回退验证，也可以显式改回 `openclaw_builtin`，但那不再是正式默认值。

自动化联调测试当前统一参考 [`tests/test_openclaw_p0_automation.py`](tests/test_openclaw_p0_automation.py)；顶层 `tests/` 已作为 ClawRadar 当前主线测试入口。

## 12. 实际选择建议

对大多数调用方，可直接按下面选：

- 想按主题发起一轮流程：用 `user_topic + full_pipeline + archive_only`
- 想从真实来源抓一轮：用 `real_source + full_pipeline` 或 `real_source + crawl_only`
- 想先人工看选题：用 `topics_only`
- 想只判断值不值得写：用 `score_only`
- 已有评分结果，想先出稿：用 `write_only`
- 已有内容包，想重投递或重留档：用 `deliver_only`
- 已有中间工件但不想手动判断从哪继续：用 `resume`

当前最推荐的一组正式默认参数是：

```json
{
  "entry_options": {
    "write": {
      "executor": "external_writer"
    },
    "delivery": {
      "target_mode": "archive_only",
      "target": "archive://openclaw_p0"
    },
    "degrade": {
      "input_unavailable": "fail",
      "write_unavailable": "fail",
      "delivery_unavailable": "fail"
    }
  }
}
```

这组参数对应当前正式边界：真实抓取、真实写作、archive-only 留档、失败显式暴露，不再把 builtin 当成默认正式写作路径。
